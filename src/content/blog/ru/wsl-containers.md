На конференции Microsoft Build 2026 разработчики анонсировали долгожданное событие: встроенную поддержку **WSL-контейнеров** (WSL Containers). Эта функция позволяет собирать, запускать и управлять Linux-контейнерами непосредственно внутри подсистемы Windows для Linux (WSL), полностью избавляя от необходимости устанавливать тяжеловесные сторонние платформы, такие как Docker Desktop.

Годами разработчики на Windows стояли перед выбором: использовать Docker Desktop и мириться с его высокой нагрузкой на систему, либо вручную разворачивать Podman или Docker-демон в режиме rootless внутри WSL. WSL-контейнеры призваны решить эту проблему, предоставляя легковесную встроенную альтернативу, которая работает прямо из коробки, не требуя дополнительных фоновых служб и не создавая сложностей с лицензированием.

## Почему Docker Desktop стал проблемой

В 2022 году Docker обновил условия лицензионного соглашения, сделав использование Docker Desktop платным для компаний с числом сотрудников более 250 или годовым доходом более 10 миллионов долларов. Это решение заставило многие корпоративные команды искать соответствующие лицензионным требованиям альтернативы.

Помимо юридических вопросов, Docker Desktop давно критикуют за ресурсоемкость. Платформа запускает собственную служебную виртуальную машину, которая даже в режиме простоя нередко потребляет от 2 до 4 ГБ оперативной памяти. Кроме того, фоновые вспомогательные процессы создают ощутимую нагрузку на процессор, ускоряя разрядку батареи ноутбуков.

## Основные компоненты системы

Архитектура WSL-контейнеров предлагает разработчикам два ключевых инструмента:

1. **Консольная утилита `wslc.exe` (с алиасом `container.exe`)** — позволяет создавать, запускать, останавливать и инспектировать контейнеры через PowerShell, CMD или терминал любой дистрибуции WSL.
2. **WSL Container API** — библиотека, доступная в виде NuGet-пакета `Microsoft.WSL.Containers`, которая позволяет разработчикам программно управлять жизненным циклом контейнеров прямо из нативных Windows-приложений.

Поскольку эти инструменты работают напрямую с ядром виртуальной машины WSL, контейнеры запускаются практически мгновенно и динамически делят аппаратные ресурсы хоста.

## Пошаговое руководство по настройке и использованию

Чтобы протестировать публичную предварительную версию WSL-контейнеров, выполните следующие действия:

### Шаг 1: Обновление WSL
Убедитесь, что у вас установлена последняя предварительная сборка WSL. Откройте PowerShell или командную строку от имени Администратора и выполните:

```bash
wsl --update --pre-release
```

*Примечание: для поддержки встроенных контейнеров требуется WSL версии 2.9.3 или выше.*

### Шаг 2: Перезапуск WSL
Перезагрузите среду подсистемы для применения изменений:

```bash
wsl --shutdown
```

### Шаг 3: Проверка CLI
Убедитесь, что консольный клиент доступен в системе:

```bash
wslc --version
```

### Шаг 4: Запуск тестового контейнера
Теперь вы можете запускать стандартные OCI-образы. Например, запустим интерактивную сессию Ubuntu:

```bash
wslc run -it --name test-box ubuntu:latest /bin/bash
```

### Шаг 5: Сборка из Dockerfile
WSL-контейнеры полностью поддерживают стандартные файлы конфигурации сборки. Создайте простой `Dockerfile` и выполните:

```bash
wslc build -t my-custom-app:1.0 .
```

## Сравнение альтернатив

| Функция / Метрика | WSL Containers | Docker Desktop | Podman (WSL) |
| :--- | :--- | :--- | :--- |
| **Лицензирование** | Бесплатно (встроено в Windows) | Платное для крупных компаний | Бесплатно (Open Source) |
| **Потребление памяти (простой)**| Минимальное (< 100 МБ) | 2 ГБ - 4 ГБ | ~150 МБ - 300 МБ |
| **Работа без демона** | Да | Нет (нужен демон) | Да (в режиме rootless) |
| **Поддержка Compose** | Базовая (через утилиты) | Полная | Полная (podman-compose) |
| **Скорость запуска** | Мгновенно (< 1 сек) | Медленно (15–45 сек) | Быстро (1–3 сек) |
| **Доступность ОС** | Только Windows | Windows, macOS, Linux | Windows, macOS, Linux |

## Сеть и монтирование томов

WSL-контейнеры наследуют сетевой стек WSL 2, что существенно упрощает проброс портов. По умолчанию любые службы, запущенные внутри контейнера, доступны на хосте через `localhost`.

### Проброс портов
Для запуска веб-сервера Nginx с пробросом порта 80 на порт 8080 хост-машины выполните:

```bash
wslc run -d -p 8080:80 --name web-server nginx:latest
```

### Монтирование директорий
Поддерживается монтирование папок как из файловой системы Windows (`/mnt/c/...`), так и из внутренней файловой системы Linux дистрибутивов WSL. Пример монтирования:

```bash
wslc run -v /home/user/app:/app -w /app node:18 npm start
```

*Совет по производительности: всегда храните исходный код проектов внутри нативной файловой системы WSL (например, в каталоге `/home/username/projects`), а не в папках Windows (вроде `/mnt/c/`), чтобы избежать снижения скорости из-за трансляции файловых вызовов между операционными системами.*

## Интеграция с VS Code и Dev Containers

Вы можете настроить редактор VS Code для использования `wslc.exe` в качестве рантайма для расширения **Dev Containers**. Для этого добавьте в глобальный файл настроек `settings.json` следующую строку:

```json
{
  "devcontainers.dockerPath": "wslc.exe"
}
```

После этого сборка контейнеров разработки будет осуществляться через встроенную подсистему WSL в обход Docker Desktop, сохраняя при этом все возможности отладки и терминала.

## Рекомендации по безопасности

Так как WSL-контейнеры работают близко к ядру WSL, важно соблюдать правила безопасности:
- **Используйте rootless-режим:** Старайтесь не запускать процессы в контейнере от имени суперпользователя. Добавляйте непривилегированного пользователя в инструкции вашего `Dockerfile`.
- **Минимизируйте образы:** Используйте легковесные базовые образы (например, Alpine или distroless), чтобы уменьшить потенциальную площадь атаки.
- **Изолируйте дистрибутивы:** Проводите эксперименты с непроверенным софтом в отдельных, специально созданных дистрибуциях WSL.

## Дорожная карта и выводы

На текущий момент функция находится в статусе публичного превью, а стабильный релиз (General Availability) запланирован на осень 2026 года. Анонсированная технология представляет собой важный архитектурный сдвиг: Microsoft интегрирует контейнеризацию непосредственно в ядро операционной системы Windows. Это дает разработчикам быстрый, легкий и юридически чистый инструмент для повседневного тестирования локальных приложений.
